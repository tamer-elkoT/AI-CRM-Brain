This table is the "brain" of your system. It translates the cold math from your ML model into the human-readable advice that your sales managers will actually use.

Here is the breakdown of what each column stores, why it is beneficial, and how to handle the table structure.

### Column Breakdown & Benefits

#### 1. Identifiers & Relationships

* **`id` (UUID):** A unique identifier for this specific recommendation.
* **`deal_id` (String/ForeignKey):** Links this advice back to the specific `zoho_deals` row. If the deal is deleted in Zoho, `CASCADE` ensures this recommendation is also deleted, keeping your database clean.
* **`prediction_id` (UUID/ForeignKey):** Links to the exact ML run (`ml_predictions`) that triggered this advice. *Benefit:* If an ML model goes rogue and gives a 99% win score to a bad deal, you can trace exactly which ML run caused the LLM to give bad advice.

#### 2. The Core AI Context

* **`llm_payload` (JSONB):** Stores the exact JSON string (deal details + ML score) that you sent to the LLM.
* *Benefit:* If the LLM "hallucinates" (makes something up), you look here. Did you accidentally send it the wrong company name, or did the LLM just invent it? This is crucial for prompt debugging.



#### 3. Scoring & Math

* **`adjusted_probability` (Numeric 5,4):** The raw decimal (e.g., `0.8521`) output by the LLM.
* **`adjusted_score_pct` (Computed):** The database automatically turns `0.8521` into `85.21`. *Benefit:* Your Streamlit UI doesn't have to do math. It just reads this column and displays "85%".
* **`base_probability` (Numeric 5,4):** A copy of the original ML model's score.
* **`score_delta` (Computed):** The difference between the ML model and the LLM's assessment. *Benefit:* The UI uses this to show "+5%" or "-10%" trend arrows.

#### 4. The Human Output

* **`recommendation_ar` & `recommendation_en` (Text):** The actual advice generated by the LLM (e.g., "اتصل بالعميل غدًا"). *Benefit:* Storing both supports your future Arabic RTL UI requirement while keeping English available for debugging.
* **`priority_tier` (Computed):** Automatically categorizes the deal as 'HIGH', 'MEDIUM', or 'LOW' based on the score. *Benefit:* Your Streamlit dashboard can instantly group deals into red, yellow, and green buckets without writing complex Python logic.
* **`risk_flag` & `risk_reasoning` (String/Text):** Categorizes specific dangers (e.g., "COMPETITOR_PRESENT").

#### 5. MLOps & System Health

* **`llm_model_id` & `prompt_version` (String):** Records which LLM (e.g., "gpt-4o") and prompt version generated this row. *Benefit:* If you upgrade your prompt next week, you can query to see if "v2.0" leads to more closed deals than "v1.0".
* **`llm_latency_ms` & `llm_tokens_used` (Integer):** Tracks how long the API call took and how much it cost. *Benefit:* Essential for monitoring your OpenAI/Anthropic API bills.

#### 6. The User Feedback Loop

* **`rep_action_taken` (Boolean), `rep_feedback_at`, `rep_feedback_text`:** Tracks if the salesperson actually clicked "I did this!" on your UI. *Benefit:* This is the ultimate metric of MVP success. If the AI gives advice but reps never mark it "Actioned," you know your prompt engineering needs work.
